## Changes from previous plan

- Aligns the model with `docs/configurable-workflows-architecture.md`: current roles should map cleanly to future capability checks, but this issue should not add the configurable workflow DSL/runtime.
- Uses existing ASP.NET Core Identity roles plus authorization policies, not a new permission storage system.
- Treats admin/configuration reads that expose setup or secret-like data as admin-only, especially GitHub App repository listing and integration details.
- Keeps workflow read access separate from operational workflow commands so read-only users can inspect runs/events/logs.

## Implementation Plan

1. **Introduce named permission policies**
   - Update `src/hhnl.Formicae.Api/ManagementAuthorization.cs`.
   - Replace the single public policy constant with:
     - `WorkflowView`
     - `WorkflowOperate`
     - `ManagementAdmin`
   - Keep `ManagementAuthorized` as a temporary compatibility alias if it reduces churn.
   - Implement one reusable authorization requirement/handler that checks a requested permission name.
   - Preserve existing bypass behavior:
     - `ManagementAuth:Enabled=false` succeeds.
     - local development bypass succeeds when configured.
   - Back policies with ASP.NET Core Identity roles:
     - `WorkflowViewer`
     - `WorkflowOperator`
     - `ManagementAdmin`
   - Role implication:
     - admin satisfies admin, operate, view
     - operator satisfies operate, view
     - viewer satisfies view only

2. **Extend `ManagementUserService` role helpers**
   - In `src/hhnl.Formicae.Infrastructure/Identity/ManagementUserService.cs`, add constants for the three roles and helpers:
     - `IsInPermissionAsync(ClaimsPrincipal principal, string permission)`
     - `GrantViewerAsync(FormicaeUser user, CancellationToken ct)`
     - `GrantOperatorAsync(FormicaeUser user, CancellationToken ct)`
     - `GrantAdminAsync(FormicaeUser user, CancellationToken ct)`
   - Ensure helper methods create missing roles via `RoleManager<IdentityRole>`.
   - Keep `AuthorizedUserRole` and `GrantAuthorizedUserAsync` as compatibility aliases, but route them to admin.
   - Keep `IsAuthorizedAsync` as a compatibility alias, but make it true for admin/operator/viewer if it is used only as “allowed into UI”; otherwise make it admin-equivalent and update callers explicitly.

3. **Update bootstrap and invite behavior**
   - In identity-provider activation inside `Program.cs`, replace `GrantAuthorizedUserAsync` with `GrantAdminAsync`.
   - In `InviteService`, make redeemed invites grant `ManagementAdmin` for compatibility with the existing “authorized users invite other users” flow.
   - Do not add database migrations for roles; Identity roles are created through existing `RoleManager`.

4. **Expose UI capabilities**
   - Update `/api/auth/current-user` in `src/hhnl.Formicae.Api/Program.cs`.
   - Return:
     - `canViewWorkflows`
     - `canTriggerWorkflows`
     - `canAdminister`
   - Keep `authorized` for compatibility during this change.
   - Recommended derivation:
     - `authorized = canViewWorkflows || canTriggerWorkflows || canAdminister`
   - When auth is disabled or dev bypass applies, return all capability booleans as `true`.

5. **Apply server-side policies by endpoint**
   - Require `WorkflowView` on workflow read endpoints:
     - `GET /api/workflows`
     - `GET /api/workflows/{workflowId}`
     - `GET /api/workflows/{workflowId}/runs`
     - `GET /api/workflows/{workflowId}/events`
     - `GET /api/workflows/{workflowId}/signals`
     - `GET /api/workflows/{workflowId}/chat-messages`
     - `GET /api/workflows/{workflowId}/logs`
   - Require `WorkflowOperate` on workflow action endpoints:
     - `POST /api/workflows/github-issue`
     - `POST /api/workflows/{workflowId}/retry`
     - `POST /api/workflows/{workflowId}/runs/{taskRunId}/retry`
   - Require `ManagementAdmin` on admin/configuration endpoints:
     - `PUT /api/ai-settings`
     - `POST /api/integrations/github`
     - `PUT /api/integrations/{integrationId}/github-app`
     - `POST /api/integrations/{integrationId}/webhook-secret`
     - `DELETE /api/integrations/{integrationId}`
     - `POST /api/integrations/{integrationId}/repositories`
     - `DELETE /api/integrations/{integrationId}/repositories/{repositoryId}`
     - `PUT /api/integrations/{integrationId}/identity-provider`
     - `POST /api/integrations/{integrationId}/identity-provider/restart`
     - `GET/POST /api/auth/invites`
     - `GET /api/auth/github/repositories`
   - Consider protecting admin reads that expose setup/admin data with `ManagementAdmin`:
     - `GET /api/integrations/{integrationId}` because it returns webhook/setup details
     - `GET /api/integrations/{integrationId}/repositories`
   - Leave worker callbacks and GitHub webhooks on their existing non-user auth mechanisms.

6. **Update frontend checks**
   - In `src/hhnl.Formicae.Api/ClientApp/src/api.ts`, extend `CurrentUser` with the three capability booleans.
   - In `src/hhnl.Formicae.Api/ClientApp/src/App.tsx`, replace broad `canMutate` with:
     - `canTriggerWorkflows` for start workflow and retry buttons
     - `canAdminister` for integrations, repositories, AI settings, identity provider controls, invite creation
     - `canViewWorkflows` for workflow loading/access checks if auth is required
   - Keep read-only users able to browse workflow details.
   - Keep invite-required gating based on `authorized`, but ensure `authorized` now means at least one permission.

7. **Add focused API tests**
   - Extend `tests/hhnl.Formicae.Tests/ManagementAuthApiTests.cs`.
   - Add factory helpers:
     - `CreateViewerAsync`
     - `CreateOperatorAsync`
     - `CreateAdminAsync`
   - Verify:
     - anonymous workflow reads return `401` when auth is enabled
     - viewer can call workflow read endpoints
     - viewer cannot start or retry workflows
     - operator can start workflow and retry workflow/task run
     - operator cannot update AI settings or mutate integrations
     - admin can update AI settings and mutate admin endpoints
     - auth-disabled mode still permits reads and mutations as before
     - `/api/auth/current-user` returns correct capability booleans for viewer/operator/admin

8. **Add focused identity tests**
   - Extend `tests/hhnl.Formicae.Tests/ManagementIdentityTests.cs`.
   - Verify:
     - grant helpers create missing roles
     - admin implies operator and viewer permissions
     - operator implies viewer but not admin
     - viewer does not imply operator/admin
     - invite redemption grants the admin role
     - identity-provider bootstrap grants the admin role

9. **Keep future workflow architecture compatibility explicit**
   - Add a short comment or small central mapping in authorization code explaining that these policies are coarse management permissions now and are intended to map later to workflow capabilities such as trigger, retry, approve version, secret access, and admin.
   - Do not implement capability persistence, workflow definition permission validation, secret scopes, or configurable workflow editor permissions in this issue.
