## Changes from previous plan

- Replace the custom `AuthorizedUser` table with ASP.NET Core Identity users, external logins, and an `AuthorizedUser` role or claim.
- Move enforcement from path-scanning middleware to named authorization policies on Minimal API endpoint groups.
- Make GitHub the first external-login provider implementation, but introduce provider-neutral user/linking helpers so future providers do not require redesign.
- Add cookie redirect handling so API callers receive `401/403` while browser UI users are redirected appropriately.

## Implementation Plan

1. **Align with ASP.NET Core Identity**
   - Keep `FormicaeDbContext : IdentityDbContext<FormicaeUser>`.
   - Extend `FormicaeUser` with provider-neutral profile fields:
     - `DisplayName`
     - `LastLoginAt`
     - `CreatedAt`
     - `UpdatedAt`
   - Use `AspNetUserLogins` for provider mappings: `LoginProvider`, `ProviderKey`, `ProviderDisplayName`.
   - Add roles support in [Program.cs](/tmp/hhnl.Formicae/src/hhnl.Formicae.Api/Program.cs:26):
     - `.AddRoles<IdentityRole>()`
     - seed or lazily create role `AuthorizedUser`.

2. **Add management auth options**
   - Add `ManagementAuthOptions` bound from `ManagementAuth`.
   - Fields:
     - `Enabled`
     - `InviteCodeExpiration`
     - `BypassForLocalDevelopment`
   - Default should preserve local ergonomics:
     - auth disabled when `ManagementAuth:Enabled=false`
     - when any identity provider is enabled and management auth is enabled, mutating management endpoints require authorization.

3. **Create provider-neutral identity service**
   - Add an application/infrastructure service such as `ManagementUserService`.
   - Responsibilities:
     - find or create Identity user from external provider login
     - call `UserManager.AddLoginAsync(...)`
     - update profile and `LastLoginAt`
     - check whether user has `AuthorizedUser`
     - grant `AuthorizedUser`
   - First provider implementation should map GitHub as:
     - provider name: `GitHub`
     - provider key: GitHub numeric user id
     - username/display name: GitHub login/name
     - email when available
   - Do not hard-code authorization logic to GitHub except inside GitHub login mapping.

4. **Update GitHub OAuth callback to use Identity**
   - In `/api/auth/github/callback`, replace direct `context.SignInAsync(...)` with:
     - exchange OAuth code as today
     - fetch GitHub user as today
     - call `ManagementUserService.FindOrCreateExternalUserAsync(...)`
     - store access token with Identity auth properties if repository browsing still needs it
     - sign in with `SignInManager.SignInAsync(user, properties)`
   - Preserve existing GitHub challenge/callback URLs.

5. **Add invite code persistence**
   - Add `InviteCode` entity and `DbSet`.
   - Store:
     - `Id`
     - `CodeHash`
     - `CreatedByUserId`
     - `UsedByUserId`
     - `CreatedAt`
     - `ExpiresAt`
     - `UsedAt`
   - Add unique index on `CodeHash`.
   - Store only a hash of a cryptographically random invite code.
   - Add EF migration and update the model snapshot.

6. **Add invite service**
   - Add `InviteService` with:
     - `CreateInviteAsync(ClaimsPrincipal creator)`
     - `RedeemInviteAsync(ClaimsPrincipal currentUser, string code)`
     - optional `ListInvitesAsync(...)`
   - Creation requires current user to already have `AuthorizedUser`.
   - Redemption requires the user to be authenticated through an external provider.
   - Redemption validates unused and unexpired code, then grants `AuthorizedUser`.

7. **Add authorization policy**
   - Add a custom policy, for example `ManagementAuthorized`.
   - Handler logic:
     - allow when `ManagementAuth.Enabled=false`
     - allow when local bypass is enabled and environment/config says local development
     - allow anonymous only for explicitly anonymous endpoints
     - otherwise require authenticated Identity user with `AuthorizedUser`
   - Configure cookie events:
     - `/api/**` unauthenticated -> `401`
     - `/api/**` forbidden -> `403`
     - browser UI unauthenticated -> GitHub challenge
     - browser UI authenticated but unauthorized -> redirect to `/?auth=unauthorized`

8. **Protect endpoint groups**
   - In [Program.cs](/tmp/hhnl.Formicae/src/hhnl.Formicae.Api/Program.cs:246), group endpoints instead of relying on `IsAnonymousPath`.
   - Require `ManagementAuthorized` for:
     - `POST /api/workflows/github-issue`
     - `PUT /api/ai-settings`
     - `POST /api/integrations/github`
     - `PUT /api/integrations/{id}/github-app`
     - `POST /api/integrations/{id}/webhook-secret`
     - `POST /api/integrations/{id}/repositories`
     - `PUT /api/integrations/{id}/identity-provider`
     - invite create/list endpoints
   - Keep anonymous or specially protected:
     - `/healthz`
     - `/api/auth/github/challenge`
     - `/api/auth/github/callback`
     - `/api/auth/current-user`
     - `/api/auth/logout`
     - `/api/auth/invites/redeem`, but require authentication
     - `/api/webhooks/github`, relying on webhook signature validation
     - static assets and fallback HTML
   - Review `/api/worker/agent-messages`; protect it with a worker callback secret/header if it must bypass user auth.

9. **Bootstrap first authorized user**
   - Change `SetIdentityProviderEnabledAsync` flow so enabling identity provider requires an authenticated current user.
   - Endpoint handler should:
     - verify caller is authenticated through the selected external provider
     - enable the identity provider
     - grant caller `AuthorizedUser` in the same operation
   - If the UI toggles identity provider while unauthenticated, redirect to GitHub login first and resume the operation after login.

10. **Update frontend**
   - Extend `ClientApp/src/api.ts` current-user response:
     - `authenticated`
     - `authorized`
     - `name`
     - `email`
     - `provider`
   - In `App.tsx`, load current user on startup.
   - Add compact auth controls:
     - login/logout
     - create invite for authorized users
     - redeem invite for authenticated unauthorized users
   - Disable or hide mutating controls when authenticated but unauthorized.
   - Keep auth-disabled local mode fully usable.

11. **Document setup**
   - Update `README.md` and deployment docs.
   - Include:
     - GitHub App callback URL: `/api/auth/github/callback`
     - `ManagementAuth` configuration
     - local auth disablement
     - first-user bootstrap behavior
     - invite workflow
     - future identity-provider expectation: external users link through Identity `AspNetUserLogins`

12. **Add focused tests**
   - Unit tests:
     - invite code generation stores hash only
     - expired/used invite rejected
     - redeeming invite grants `AuthorizedUser`
     - external provider login creates or updates Identity user
   - API tests:
     - anonymous mutating request returns `401` when auth enabled
     - authenticated unauthorized request returns `403`
     - authorized user can mutate
     - auth disabled permits local mutating calls
     - enabling identity provider grants current user authorization
     - invite redemption authorizes a second external user
   - Run:
     - `dotnet test tests/hhnl.Formicae.Tests/hhnl.Formicae.Tests.csproj`
     - `npm run build` from `src/hhnl.Formicae.Api/ClientApp`

Relevant Microsoft guidance checked: ASP.NET Core Identity supports external login providers and manages users, roles, claims, and tokens; Minimal APIs support standard authentication/authorization policies; policy-based authorization is the right fit for custom access checks.
