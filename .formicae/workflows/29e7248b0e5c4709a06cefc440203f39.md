```text
## Changes from previous plan

- Move branch-name generation and “this branch should link to this work item” intent into the application abstraction, instead of hiding it inside the GitHub provider.
- Keep GitHub’s `createLinkedBranch` as one provider implementation detail, so future source-control integrations can implement equivalent linked-branch behavior without changing workflow orchestration.

## Implementation Plan

1. Update the source-control abstraction:
   - In `src/hhnl.Formicae.Application/Workflows/Interfaces.cs`, add a request model, for example:
     - `RepositoryUrl`
     - `BaseBranch`
     - `BranchName`
     - `LinkedWorkItemUrl`
   - Change `ISourceControlProvider.CreateBranchAsync(...)` to accept that request plus `CancellationToken`.
   - Keep the model provider-neutral. Do not mention GitHub, issues, GraphQL, or node ids in the application layer.

2. Update workflow orchestration:
   - In `WorkflowOrchestrator`, generate the deterministic branch name before calling source control: `formicae/{workflow.Id:N}`.
   - Pass `workflow.RepositoryUrl`, `workflow.BaseBranch`, generated `BranchName`, and `workflow.IssueUrl` as the linked work item URL.
   - Keep existing behavior that skips branch creation when `workflow.BranchName` is already set.

3. Update non-GitHub implementations and test doubles:
   - Update `FakeSourceControlProvider` and `MockDevOpsAdapter` to accept the new request object.
   - Return `request.BranchName`.
   - Adjust captured test records so tests can assert the linked work item URL is passed through the abstraction.

4. Implement GitHub linked branch creation:
   - In `GitHubSourceControlProvider.CreateBranchAsync`, parse `request.RepositoryUrl` and `request.LinkedWorkItemUrl`.
   - Validate the linked work item belongs to the same owner/repository.
   - Fetch the base ref with `GetReferenceAsync(owner, repo, $"heads/{request.BaseBranch}")`.
   - Call `IGitHubApi.CreateLinkedBranchAsync(owner, repo, issueNumber, baseSha, request.BranchName, cancellationToken)`.
   - Preserve idempotency: if GitHub reports the linked branch already exists, return `request.BranchName`.
   - Preserve fallback: for unexpected non-cancellation linked-branch failures, create the plain git ref and return `request.BranchName`.

5. Add or update GitHub API wrapper:
   - In `OctokitGitHubApi`, ensure `IGitHubApi.CreateLinkedBranchAsync(...)` uses GraphQL:
     - Query the issue node id.
     - Run `createLinkedBranch(input: { issueId, oid, name })`.
     - Return the created ref name, falling back to the requested branch name if omitted.

6. Add focused tests:
   - Orchestrator passes `BranchName` and `LinkedWorkItemUrl` through the source-control abstraction.
   - GitHub provider calls linked-branch creation with owner, repo, issue number, base SHA, and branch name.
   - Existing linked branch returns the deterministic branch name without failing.
   - Unexpected linked-branch failure falls back to plain git ref creation.
   - Linked work item URL from another repository throws `ArgumentException`.

7. Run verification:
   - `dotnet test tests/hhnl.Formicae.Tests/hhnl.Formicae.Tests.csproj`
   - If clean, run `dotnet test hhnl.Formicae.slnx`
```
