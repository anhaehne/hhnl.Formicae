## Implementation Plan

1. Create `docs/realtime-communication-architecture.md`
   - State status: planned architecture, no implementation included.
   - Summarize current runtime:
     - Browser uses REST polling against ASP.NET Core API.
     - API owns workflow orchestration and persistence.
     - Worker is an ephemeral Kubernetes Job.
     - Worker already sends HTTP callbacks to `/api/worker/agent-messages`.
     - API uses `WorkflowTickNotifier` to wake orchestration.

2. Add a “Recommendation” section
   - Recommend **ASP.NET Core SignalR** for Server -> Browser realtime workflow updates.
   - Keep REST endpoints as the authoritative read model.
   - Browser flow:
     - Fetch current workflow snapshot through REST.
     - Open SignalR connection.
     - Join workflow-scoped groups such as `workflow:{workflowId}`.
     - On events, either patch local state or refetch affected workflow data.
   - Recommend **HTTP callbacks** for Worker -> Server communication.
   - Treat worker callbacks as durable ingestion events, not browser-specific messages.

3. Define the realtime event contract
   - Add a shared envelope shape:
     - `eventId`
     - `sequence`
     - `workflowId`
     - `taskRunId`
     - `type`
     - `occurredAt`
     - `payload`
   - Include expected event types:
     - `workflow.updated`
     - `taskRun.updated`
     - `workflowLog.appended`
     - `agentMessage.appended`
     - `workflowSignal.raised`
   - Require events to be emitted only after successful persistence.

4. Document Server -> Browser design
   - Proposed future components:
     - `WorkflowUpdatesHub`
     - workflow group subscription endpoint/method
     - authorization using existing `WorkflowView` policy
   - Cover reconnect behavior:
     - Client reconnects.
     - Refetches latest REST snapshot.
     - Resubscribes to workflow group.
   - Document scale-out requirement:
     - Single API replica works without backplane.
     - Multi-replica deployments require sticky sessions plus a SignalR backplane or managed SignalR service.

5. Document Server -> Worker design
   - Keep API-created Kubernetes Jobs as the Server -> Worker command path for now.
   - Keep worker HTTP callback as Worker -> Server status/message path.
   - Recommend future hardening:
     - HMAC signature instead of static shared header.
     - timestamp and replay window.
     - event id / sequence number for idempotency.
     - bounded payload sizes.
     - retry with backoff.
     - batching or throttling noisy stdout.
   - State that SignalR/WebSockets are not recommended for ephemeral workers unless interactive bidirectional control becomes a concrete requirement.

6. Add alternatives section
   - **SSE:** simpler browser push, but weaker for bidirectional/group-based future needs.
   - **Raw WebSockets:** more protocol ownership with little benefit over SignalR.
   - **gRPC/gRPC-Web:** strong typed contracts, but heavier for browser workflow notifications.
   - **Message broker:** RabbitMQ/NATS/Redis Streams/Kafka for future scale, replay, backpressure, or multi-service fanout.
   - **Kubernetes log watch:** useful fallback/debug tool, not primary communication contract.

7. Add reliability and persistence section
   - REST snapshots remain recovery source if realtime events are missed.
   - Prefer an outbox/event table before pushing to SignalR if exactly-once-ish delivery becomes important.
   - Do not rely on `TaskRun.Output` as the long-term event stream.
   - Consider a dedicated workflow message/event table for append-only realtime/audit data.

8. Add security section
   - Browser hub subscriptions must require authenticated users with workflow read permission.
   - Worker callbacks must authenticate independently from browser/user auth.
   - Never send secrets in realtime payloads.
   - Apply the same redaction rules used for logs, prompts, and task output.

9. Add testing guidance
   - Architecture-only issue: verify with documentation review.
   - Future implementation tests should cover:
     - event envelope serialization.
     - event emission after persistence.
     - hub authorization and group subscription.
     - reconnect/refetch behavior.
     - worker callback idempotency.
     - rejected stale or invalid worker signatures.

10. Update `README.md`
   - Add a short link under Architecture to the new realtime communication architecture document.

11. Validate docs

   - Otherwise manually check headings, links, and terminology.
   - No application tests are required because this issue has no code implementation.
