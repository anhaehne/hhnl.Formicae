```text
## Changes from previous plan

- Aligns the first slice with `docs/configurable-workflows-architecture.md`: immutable definition versions, run reproducibility, schema-first validation, and default MVP as a stored version snapshot.
- Narrows v1 runtime scope to a sequential built-in-task graph; explicitly defers typed data-flow, registry snapshots, arbitrary tasks, loops, branches, parallelism, editor drafts, and custom execution.
- Adds a small application service boundary instead of putting definition CRUD directly in stores/API code.

## Implementation Plan

1. Add v1 workflow definition models in `src/hhnl.Formicae.Application/Workflows/WorkflowModels.cs`.
   - Add `WorkflowDefinition`, `WorkflowDefinitionVersion`, and `WorkflowDefinitionStep`.
   - Use immutable version rows with: `Id`, `WorkflowDefinitionId`, `Version`, `DslSchemaVersion`, `IsEnabled`, `IsDefault`, `DefinitionJson`, `CreatedAt`.
   - Keep the v1 definition JSON sequential and schema-first:
     - `schema: "formicae.workflow/v1alpha1"`
     - `startStepId`
     - `steps[]`
     - each step: `id`, `uses`, optional `nextStepId`, optional `displayName`
   - For this issue, allow only built-in `uses` values that map to existing `TaskRunKind`: `builtins.plan`, `builtins.implement`, `builtins.create-pull-request`, `builtins.address-comments`.
   - Add `Workflow.WorkflowDefinitionId`, `Workflow.WorkflowDefinitionVersionId`, and `Workflow.DslSchemaVersion`.

2. Add a default MVP definition factory.
   - Create `DefaultWorkflowDefinitions.CreateMvp()` in the Application workflow folder.
   - Represent:
     - `plan -> implement -> createPullRequest -> addressComments`
   - Mark it as default/enabled.
   - Use stable identifiers for the built-in definition where practical, but generated version IDs are acceptable if seeding can resolve by definition key/name.

3. Add validation.
   - Create `WorkflowDefinitionValidator`.
   - Validate:
     - non-empty definition name
     - supported schema version
     - non-empty steps
     - unique step ids
     - valid `startStepId`
     - every `nextStepId` points to an existing step
     - no cycles in the sequential graph
     - all steps reachable from `startStepId`
     - exactly one terminal step for v1
     - every `uses` maps to a supported built-in task
   - Return structured validation errors instead of throwing for normal invalid input.

4. Add definition store contracts in `Interfaces.cs`.
   - Add methods to:
     - list definitions
     - get definition by id
     - create definition
     - create immutable version
     - get definition version by id
     - get latest enabled version for a definition
     - get default enabled version
   - Keep these on `IWorkflowStore` for a small slice, unless the implementation naturally justifies a separate `IWorkflowDefinitionStore`.

5. Implement definition persistence in memory.
   - Extend `InMemoryWorkflowStore` with dictionaries/lists for definitions and versions.
   - Seed the default MVP definition lazily or through an explicit `EnsureDefaultWorkflowDefinitionAsync`.
   - Ensure tests using a fresh in-memory store can start workflows without manual seeding.

6. Implement EF persistence.
   - Add `DbSet<WorkflowDefinition>` and `DbSet<WorkflowDefinitionVersion>` to `FormicaeDbContext`.
   - Configure tables:
     - `workflow_definitions`
     - `workflow_definition_versions`
   - Store the canonical v1 graph in `DefinitionJson`.
   - Add indexes for default lookup and `(WorkflowDefinitionId, Version)`.
   - Add nullable workflow columns first if needed for existing rows: `WorkflowDefinitionId`, `WorkflowDefinitionVersionId`, `DslSchemaVersion`.
   - Add an EF migration and update the snapshot.

7. Add default seeding for EF.
   - Add a startup seeding path in API startup or infrastructure dependency initialization.
   - If the default definition/version does not exist, create it.
   - Validate the generated MVP definition before saving it.
   - Do not require existing deployments to manually create the default before workflows can run.

8. Add an application service for definitions.
   - Create `WorkflowDefinitionService`.
   - Responsibilities:
     - create definitions
     - create versions
     - list/read definitions
     - validate before saving enabled versions
     - resolve default/latest/requested enabled version for run creation
   - Keep authorization in API, not in this service.

9. Update workflow creation.
   - Extend `StartGitHubIssueWorkflowRequest` with optional `WorkflowDefinitionId` and `WorkflowDefinitionVersionId`.
   - In `WorkflowService.StartGitHubIssueWorkflowAsync`:
     - resolve requested version if supplied
     - otherwise resolve default MVP version
     - validate the selected version before creating the run
     - persist definition/version/schema ids on `Workflow`
   - Return `400` for invalid requested versions and `404` for missing requested versions at the API boundary.

10. Keep orchestration compatible.
   - Do not rewrite `WorkflowOrchestrator` into a generic scheduler in this issue.
   - Add a small helper that loads the workflow’s definition version and confirms the current static transition/task exists in that graph before scheduling built-in task runs.
   - If validation fails at runtime, fail the workflow before starting an agent or creating a PR.
   - Preserve all existing behavior for label gates, plan feedback reruns, PR creation, and comment addressing.

11. Add management API endpoints in `Program.cs`.
   - `GET /api/workflow-definitions`
   - `GET /api/workflow-definitions/{id}`
   - `POST /api/workflow-definitions`
   - `POST /api/workflow-definitions/{id}/versions`
   - Use existing management authorization.
   - Return validation errors as `400` with a clear error list.

12. Add focused tests.
   - Validator accepts the default MVP graph.
   - Validator rejects duplicate step ids.
   - Validator rejects missing start step.
   - Validator rejects unknown `nextStepId`.
   - Validator rejects cycles.
   - Validator rejects disconnected steps.
   - Validator rejects unsupported `uses`.
   - Starting a workflow without definition fields uses the default MVP version.
   - Starting a workflow with a stored custom sequential built-in definition stores that version on the run.
   - Starting with a missing version returns `404` or service-level not-found result.
   - Default MVP workflow still completes through existing orchestrator tests.

13. Update docs.
   - Update `docs/mvp-workflow.md` to say the MVP workflow is now represented by a stored default definition version while execution remains the existing built-in path.
   - Update `docs/configurable-workflows-architecture.md` with an “Implemented v1 subset” note:
     - sequential built-in task graph only
     - immutable persisted versions
     - validation before save/run
     - no typed data-flow, registry snapshot, custom tasks, loops, branches, parallelism, UI editor, or arbitrary script execution yet.

14. Run verification.
   - `dotnet test hhnl.Formicae.slnx`
   - Inspect the EF migration and model snapshot for:
     - new definition tables
     - workflow definition/version/schema columns
     - expected indexes
   - Run existing workflow orchestration tests to confirm the default MVP path did not regress.
```
