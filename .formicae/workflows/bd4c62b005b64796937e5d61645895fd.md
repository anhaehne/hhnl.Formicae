```text
## Changes from previous plan

- Keep authentication opt-in everywhere, including raw Kubernetes manifests and Helm. Do not enable auth by default in chart values.
- Make protected mode conditional on `Auth:Enabled=true`; existing local and default Kubernetes deployments should continue working anonymously unless configured otherwise.
- Split work into smaller backend, persistence, frontend, deployment, and docs steps with tests after each behavioral change.

## Implementation Plan

1. **Prepare API integration testing**
   - Add `public partial class Program;` to `src/hhnl.Formicae.Api/Program.cs`.
   - Add `Microsoft.AspNetCore.Mvc.Testing` to `tests/hhnl.Formicae.Tests`.
   - Create a reusable `WebApplicationFactory<Program>` test fixture that can override config values such as `Auth:Enabled`.

2. **Add auth configuration model**
   - Add `AuthOptions` under the API or Application layer:
     - `Enabled`
     - `Provider`
     - `CookieName`
     - `GitHub:ClientId`
     - `GitHub:ClientSecret`
     - `AllowedGitHubLogins`
     - `AllowedEmails`
     - `InviteCodes`
   - Default `Auth:Enabled=false` in `appsettings.json` and `appsettings.Development.json`.
   - Treat missing GitHub OAuth settings as startup errors only when `Auth:Enabled=true`.

3. **Add user authorization persistence**
   - Add an `AuthUser` entity with:
     - `Id`
     - `GitHubUserId`
     - `GitHubLogin`
     - `Email`
     - `DisplayName`
     - `InviteCodeHash` or invite code id
     - `CreatedAt`
   - Add `IAuthUserStore` in Application.
   - Add `InMemoryAuthUserStore` for fake/local mode.
   - Add `EfAuthUserStore` and wire it into `FormicaeDbContext`.
   - Add an EF migration for the `auth_users` table.
   - Register the correct store in `AddFormicaeInfrastructure`.

4. **Implement invite validation service**
   - Add `AuthInviteService`.
   - Support static configured invite codes for MVP.
   - Hash invite codes before comparing/storing accepted codes.
   - Implement:
     - validate configured invite code
     - accept invite for current GitHub identity
     - check whether a GitHub identity is allowed through allow-list or accepted invite
   - Unit test valid invite, invalid invite, and previously accepted user.

5. **Add cookie + GitHub OAuth authentication**
   - Add required ASP.NET Core auth packages if not already available.
   - In `Program.cs`, when `Auth:Enabled=true`:
     - call `AddAuthentication`
     - configure cookie auth using `Auth:CookieName`
     - configure GitHub OAuth using:
       - authorization endpoint `https://github.com/login/oauth/authorize`
       - token endpoint `https://github.com/login/oauth/access_token`
       - user endpoint `https://api.github.com/user`
   - Store GitHub id, login, name, and email claims in the cookie.
   - On OAuth success, allow sign-in only if the user is allow-listed or has accepted an invite.
   - When `Auth:Enabled=false`, do not require or configure OAuth credentials.

6. **Add auth/session endpoints**
   - Add endpoints:
     - `GET /api/auth/session`
     - `GET /api/auth/login?returnUrl=/`
     - `POST /api/auth/logout`
     - `POST /api/auth/invite/accept`
   - `session` should return:
     - `authEnabled`
     - `authenticated`
     - `allowed`
     - `login`
     - `name`
     - `email`
   - Keep `session`, `login`, and `invite/accept` anonymous.
   - Require auth for `logout`.
   - `invite/accept` should require a signed-in GitHub identity but may return `401` if no OAuth cookie exists.

7. **Add conditional authorization policy**
   - Add a policy such as `RequireManagementAuth`.
   - When `Auth:Enabled=true`, require authenticated and allowed users.
   - When `Auth:Enabled=false`, allow anonymous access.
   - Apply the policy to mutating endpoints:
     - `POST /api/workflows/github-issue`
     - `PUT /api/ai-settings`
   - Do not add cookie auth to `POST /api/webhooks/github`; keep webhook signature validation as its protection.

8. **Test protected API behavior**
   - Add integration tests:
     - with `Auth:Enabled=false`, anonymous `POST /api/workflows/github-issue` reaches normal request validation/service behavior.
     - with `Auth:Enabled=true`, anonymous `POST /api/workflows/github-issue` returns `401` or `403`.
     - with `Auth:Enabled=true`, anonymous `PUT /api/ai-settings` returns `401` or `403`.
     - `POST /api/webhooks/github` still reaches the existing webhook validation path.
   - Add a test auth handler only for tests that need an authenticated allowed user.

9. **Update frontend API client**
   - Add API helpers in `ClientApp/src/api.ts`:
     - `getAuthSession`
     - `login`
     - `logout`
     - `acceptInvite`
   - Include `credentials: "same-origin"` on API calls so cookie auth works.
   - Preserve existing behavior when `authEnabled=false`.

10. **Update frontend UI flow**
   - In `App.tsx`, load auth session on startup.
   - If `authEnabled=false`, render the current app unchanged.
   - If auth is enabled and unauthenticated, show a compact GitHub sign-in screen.
   - If authenticated but not allowed, show invite-code entry.
   - Add logout control when authenticated.
   - Convert `401/403` from workflow start and AI settings update into a clear sign-in/permission error.

11. **Update Helm chart with opt-in auth**
   - In `deploy/helm/formicae/values.yaml`, add auth values with auth disabled by default:
     - `config.authEnabled: "false"`
     - `config.authProvider: "GitHub"`
     - `config.authCookieName: "formicae_auth"`
     - `config.authGitHubClientId: ""`
     - optional allow-list values
     - `secrets.authGitHubClientSecret: ""`
     - `secrets.authInviteCodes: ""`
   - In Helm `configmap.yaml`, emit:
     - `Auth__Enabled`
     - `Auth__Provider`
     - `Auth__CookieName`
     - `Auth__GitHub__ClientId`
     - allow-list config if used
   - In Helm `secret.yaml`, emit:
     - `Auth__GitHub__ClientSecret`
     - `Auth__InviteCodes`
   - Do not fail Helm rendering when these are empty unless `authEnabled` is true.

12. **Update raw Kubernetes manifests with opt-in auth**
   - In `deploy/kubernetes/base/configmap.yaml`, add auth keys with `Auth__Enabled: "false"`.
   - In `deploy/kubernetes/base/secret.example.yaml`, add example keys for:
     - `Auth__GitHub__ClientSecret`
     - `Auth__InviteCodes`
   - Keep the base deployment anonymous by default.

13. **Document setup**
   - Add `docs/authentication.md` and link it from `README.md`.
   - Document:
     - auth is opt-in
     - local dev with `Auth:Enabled=false`
     - creating a GitHub OAuth app
     - callback URL format, likely `/signin-github`
     - local protected-mode setup with user secrets or env vars
     - Helm values example
     - raw Kubernetes ConfigMap/Secret example
     - invite code setup and rotation guidance

14. **Final verification**
   - Run:
     - `dotnet test hhnl.Formicae.slnx`
     - from `src/hhnl.Formicae.Api/ClientApp`: `npm install` if needed, then `npm run build`
   - Manually smoke test:
     - default local run works with anonymous access
     - protected mode rejects anonymous workflow start
     - GitHub login flow reaches OAuth challenge when configured
     - invite acceptance makes the signed-in user allowed.
```
